Rubric

topics

Python functions, return values, string parsing, conditionals, importing modules

further study concepts

N/A

requires

Calculator 1 solution, basic command line

time

1-2 hours

authors

Christian Fernandez (updates by Joel Burton)

Goals

Students will draw on their solution to the previous calculator exercise and extend the functionality to be more calculator-like. They will learn to use functions imported from other files, check and/or adapt user input, utilize for-loops and conditionals.

Gotchas

  1. Handle the case where a user enters only one number

  2. Handle the case where inputs are not numbers

  3. Dividing with floats

  4. Typing something besides an operator as the first character

  5. Handles more than one space between operators or numbers

  • If the student’s file isn’t running the changes they have made, make sure they’re editing the correct version (it can get confusing because they have the HB version and their version from calc-1)

Code Reviews

  • A large part of this assignment is handling bad user input. Things to check for:

    • The program deals with only one num input or inputs are not numbers.

    • The program handles an invalid operator.

    • The program can handle more than one space.

  • A common question is how to deal with multiple numerical inputs for operators like “pow”. Have them make a decision if they want to just evaluate the first number or print all of the powers of the input.

Code Review 1

  • Review the idea of importing methods from another file

  • Students evaluate all functions in arithmetic.py.

  • Make sure the students know how break and continue work.

  • Any checks for input sanitation should be before the calculations.

  • Check that the program divides with floats, so no floor division

  • Make sure the students know the control flow of this program; look at the if-elif-else statements

Further Study #1

  • Students are asked to make a new arithmetic.py with functions that take in a list of number values. We’re trying to steer them away from using *args.

Further Study #2

  • Students try to break the calculator by typing in different types of bad inputs and then fixing them so that the calculator doesn’t break but instead handles the errors. The goal is to get the calculator to not error out when there’s bad inputs.

  • Discuss how they were able to break the calculator and what they did to handle it.

  • Potential edge cases:

    • What happens when you type in a word instead of a number

    • Typing a command incorrectly (the inputs are out of order)

    • Typing in too few or too many numbers for a command.

Further Study #3

  • Using reduce() in calculator.py to take in any number of inputs. Students should have already updated arithmetic.py to process a list of number inputs.

  • Chat about the documentation, how they worked through it and understood what reduce() does. Ask for a few examples in the terminal/walk through their code.

  • You might also talk about map() and filter(), which do similar things as reduce(). Note that map() and filter() now return iterators that need to be wrapped in list() to get a list.

Further Study #4

  • Students should create their own reduce() function (called my_reduce()) that replicates the functionality of the built-in reduce().

Further Study #5

  • Customize the number of decimal places the calculator should show. This is open to interpretation – do users pass in a number of decimals and there’s a default if nothing is passed in? Or should they set every number to only display a certain number of decimal points?

Further Study #6

  • Have calculator.py read from a file of operations.

    • Print the results when the file is run.

    • Write the results to a new file.

Further Study #7

  • General discussion about how code could be improved. Is it readable, clean, DRY? Good variable names? Docstrings?
